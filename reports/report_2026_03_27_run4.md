## Recommendations

### 🔴 P0 — Scout Latency (Critical)

**Problem**: Scout median latency is 6.8s, 3.4× above the 2s target. 14% of calls exceed 10s.

**Impact**: Slow Scout delays the entire pipeline. At 7s per Scout call × 9 symbols = 63s just for Scout tier per cycle. This means the system can only run ~1 cycle per minute even at 60s sleep.

**Recommendation**: Investigate the OpenRouter routing. The Scout is supposed to use Haiku/DeepSeek-class models (~500ms target). Possible causes:
- Model routing may be hitting a slower model
- Context window may be too large (check Scout prompt token count)
- OpenRouter rate limiting or queue delays
- Consider switching to a direct API endpoint instead of OpenRouter for Scout calls

### 🟡 P1 — Trader Latency Optimization

**Problem**: Trader median is 33s, max 57s. This is Opus-tier so some latency is expected, but the reasoning output suggests unnecessary verbosity.

**Recommendation**:
- Add a max_tokens constraint to the Trader prompt to cap response length
- Consider if the Trader reasoning can be streamlined — the multi-paragraph analyses are thorough but the same conclusion could be reached with fewer tokens
- For WAIT decisions (87.5% of Trader calls), consider a fast-exit path: if confidence is <0.5 after initial assessment, return WAIT immediately without full analysis

### 🟡 P1 — Scout Filter Rate Below Target

**Problem**: 77.4% filter rate vs 85-95% target. The Scout is letting 42 symbols through to the Analyst when ~28 would be ideal.

**Recommendation**:
- Tighten the volume threshold: currently many symbols pass with vol_ratio 0.7-0.9. Consider raising the INTERESTING threshold to vol_ratio ≥ 0.85 or adding a secondary filter
- Add price change threshold: require at least 0.3% price change in the recent window to escalate
- The fact that 26/42 Analyst calls returned confluence <7 suggests the Scout could catch these earlier

### 🟢 P2 — Position Limit Impact

**Problem**: Max positions (2/2) blocked the SOLUSDT SHORT — which would have been the most profitable trade of the session (SOL dropped ~5%).

**Recommendation**:
- Consider increasing max_positions to 3 for diversified exposure
- Alternatively, implement a "displacement" mechanism: if a new signal has higher confidence than the worst-performing open position, consider closing the weak one
- The current 2-position limit cost the system its best opportunity in this session

### 🟢 P2 — XRP R:R Ratio

**Problem**: XRPUSDT SHORT had an R:R of only 1.04:1 (SL $1.3415, TP $1.3230 from entry $1.33240). This is below the typical 1.5:1 minimum for professional setups.

**Recommendation**:
- Review the Trader's SL/TP calculation for XRP — the risk/reward should be at least 1.5:1
- Consider having the Risk Validator reject trades with R:R below a configurable threshold (e.g., 1.3:1 minimum)

### 🟢 P2 — SOLUSDT Risk Validation Rejection Reason

**Problem**: The SOL rejection logged `valid=False rejections=1 size=0` but the specific rejection reason (max positions) is not in the log output. This makes debugging harder.

**Recommendation**: Log the rejection reason explicitly, e.g., `Risk validation rejected SOLUSDT: max_positions_exceeded (2/2)`.

### 🟢 P3 — Cooldown After Trade Execution

**Problem**: After DOGEUSDT SHORT was opened at 15:42, DOGE was on cooldown for 8 cycles. Meanwhile, the system continued evaluating DOGE signals but couldn't act. The cooldown period seems long relative to the session length.

**Recommendation**: Make post-trade cooldown duration configurable and consider shortening it for active symbols — the system already has an open position so further analysis of the same symbol adds no value.
